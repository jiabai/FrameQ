# FrameQ Web 宣传独立站设计规范

## 状态

Draft。本文档用于定义未来 FrameQ Web 宣传独立站的公开设计面；它不替代
`docs/DESIGN.md`，桌面客户端设计规范仍以 `docs/DESIGN.md` 为准。

设计输入与优先级：

1. 本文档是 Web 宣传站的 FrameQ 产品与体验规范。
2. [`design-system/apple-DESIGN.md`](../../design-system/apple-DESIGN.md) 是视觉风格分析和灵感参考，不是 Apple 官方规范，也不得覆盖 FrameQ 的产品事实、品牌边界或可访问性要求。
3. [`design-system/globals.css`](../../design-system/globals.css) 是未来宣传站可采用的初始设计变量，不属于当前桌面 App 的生产样式，也不会在运行时被桌面端加载。

如果三者出现冲突，以本文档、`docs/DESIGN.md` 以及产品规格中更具体的 FrameQ 约束为准。开始实现宣传站时，应在 active ExecPlan 中确认最终 token、字体授权、素材来源和组件映射。

## 目标

宣传站应帮助新访客快速理解 FrameQ、信任它的本地优先工作流，并决定是否下载、
关注或进一步了解产品。它是公开获客和产品教育界面，不是桌面客户端 UI 的网页复刻。

主要任务：

- 说明 FrameQ 是一个把公开视频链接转成本地文字稿和 AI 思考产物的桌面工具。
- 让“本地优先”的隐私边界在首屏就清晰可见。
- 用真实产品截图、录屏或忠实的产品视觉展示核心工作流。
- 在对应能力存在后，提供下载、版本说明、价格/账号和支持入口。
- 避免承诺未支持的平台、私有视频访问、Cookie 登录、云端全流程处理或安装包内置
  ASR 模型权重。

## 与桌面端设计的关系

- 宣传站可以沿用 FrameQ 的品牌色、语气、图标风格和产品语言，但不应照搬 macOS
  sheet、task monitor 或高密度桌面工作区布局。
- 桌面端保持工具优先、任务优先；宣传站可以更具叙事性和视觉表达，但仍应克制、
  具体、可信。
- 公开文案必须和桌面端产品事实一致：视频、音频和文字稿默认留在本地；可选 AI 整理
  只有在用户确认后，才可能调用管理员配置的云端 LLM。
- 宣传截图不得展示虚假的私有数据、虚假的授权状态、隐藏管理员能力或未支持的视频来源。

## 目标用户

主要用户：

- 经常从公开视频中整理信息的知识工作者、创作者、研究者、运营者和学习者。

次要用户：

- 评估本地优先视频转文字工作流的小团队。
- 查找下载、更新、文档或支持信息的现有用户。

设计应尊重这类访客的典型心理：重视隐私、时间有限，并且会警惕那些看起来会把一切上传
到云端的 AI 工具。

## 品牌气质

FrameQ 应呈现为：

- 本地优先、精确、安静但有能力。
- 面向真实工作的工具，而不是新奇演示。
- 对边界、失败场景和平台限制保持诚实。
- 现代、精致，但不装饰过度。

避免：

- 泛泛的 AI 口号、发光抽象渐变、夸张效率承诺或魔法式前后对比。
- 暗示可以绕过登录、验证码、会员、私有或年龄限制的视频访问。
- 用氛围图遮住真实产品能力。

## 信息架构

首版建议包含：

- 首页
  - 首屏：产品名、简短品类描述、本地优先信任语、主下载按钮、次级“查看工作流”按钮和真实产品视觉。
  - 工作流：粘贴公开链接、提取本地媒体、本地转写、可选 AI 总结和灵感生成。
  - 本地优先说明：哪些数据留在本地、哪些数据可能离开本机、何时需要用户确认。
  - 输出物说明：视频/音频文件、文字稿、Markdown 总结、思维导图文件、灵感结果。
  - 支持来源说明：说明当前支持的公开链接范围和已知限制。
  - 下载区：平台可用性、版本、安装包大小提示、首启模型下载说明和 release notes 入口。
  - FAQ/支持：模型缓存、日志、网络失败、公开视频限制和联系路径。
- 下载页（需要时）
  - 仅当对应发行产物真实存在时，分别展示 macOS Apple Silicon、macOS Intel 和 Windows。
  - 清楚说明 ASR 模型在首启下载，不内置在安装包中。
- 隐私页（需要时）
  - 用普通语言解释数据流，并与产品和安全文档保持一致。

## 首屏规则

首屏必须让访客无需滚动就理解产品。

必备信号：

- `FrameQ` 作为最醒目的品牌/产品名。
- 直接的品类描述，例如 `视频转文字与 AI 整理桌面工具`。
- 靠近主按钮的本地优先信任语。
- 指向当前有效下载、候补名单或了解入口的主 CTA。
- 真实产品截图、短录屏或忠实的产品视觉；避免泛用抽象 hero 图。

首屏在桌面和移动端都应露出下一屏内容的提示，不要做成整屏封闭的视觉海报。

## 视觉系统

- 颜色应专业克制：中性背景、清晰深色文字、主操作使用 FrameQ 蓝，本地保存/成功使用
  克制绿色，错误和警告才使用红色。
- 避免单调的蓝紫渐变主题、装饰光球、玻璃堆叠和大面积空洞营销卡片。
- 产品图应真实、清晰、可读，能传达工作流；不要把截图模糊到无法识别。
- 可以用留白提升公开阅读体验，但内容必须具体、可扫读。
- 卡片只用于重复功能、输出物、FAQ 或下载项；不得卡片套卡片。
- 图标用于辅助扫读，需保持同一线条风格；含义必须由文字标签承载，不能只靠图标。

## 字体与文案

- 标题应直接、具体、和产品强相关；优先使用清楚的名词和动词，避免抽象 AI slogan。
- 正文短、平实、准确，限制条件要直接说明。
- 中文是主要文案语言；ASR、Markdown、LLM、Mermaid 等术语在更精确时可保留英文。
- 不用可见说明文字解释显而易见的网页操作；通过标签、层级和组件状态表达。

建议文案层级：

- 产品名：`FrameQ`
- 品类：`视频转文字与 AI 整理桌面工具`
- 信任语：`默认本地处理视频、音频和文字稿；AI 整理需你确认后才会调用云端 LLM。`
- 工作流语：`粘贴公开链接，保存本地文字稿，再生成可继续思考的总结和灵感。`

## 组件

首版使用少而稳定的组件：

- Header：品牌、产品/文档/下载/支持链接、主下载动作。
- Hero：产品名、品类、信任语、CTA 组、产品视觉。
- Workflow steps：4 到 5 个紧凑步骤，并有清晰状态标签。
- Trust boundary panel：本地数据、可选云端数据和绝不包含的数据。
- Output preview：文字稿、总结、思维导图文件和灵感示例。
- Download card：平台、架构、版本、文件类型、首启模型说明。
- FAQ：问题足够简短时可用 accordion；否则用简单列表。
- Footer：release notes、隐私、支持、仓库或公司链接。

不要在产品和商业面未定义前加入聊天浮窗、紧迫感横幅、虚假证言或价格承诺。

## 响应式规则

- 桌面端使用受控内容宽度；只有当对比能提升理解时才使用双列区块。
- 平板端在产品视觉变拥挤时，将视觉堆叠到文案下方。
- 移动端保持 CTA 可见，产品截图可检查；除代码块/表格等明确可滚动内容外，避免横向滚动。
- 导航必须稳定、紧凑，不要把关键下载/支持动作隐藏在非典型手势里。

## 可访问性

- 正文、说明文字和 disabled 状态都必须保持可读对比度。
- 标题层级使用语义化顺序。
- 按钮和链接必须有可见 focus 状态和明确 accessible name。
- 产品截图旁必须有文字解释，不依赖图片内文字传达关键信息。
- 尊重 reduced motion；工作流动画在用户减少动态效果时应暂停或简化。

## 性能与 SEO

- 首屏和核心页面优先静态化或服务端渲染。
- 截图和视频必须有响应式尺寸、压缩版本和 poster 图。
- 页面需要清晰的 title、meta description、Open Graph 图和结构化内容标题。
- 未形成隐私决策前，不加载 analytics、追踪脚本或外部嵌入。
- 下载链接和 release metadata 必须来自正式发布流程，不能硬编码猜测值。

## 内容安全与产品事实

- 不得暗示 FrameQ 能访问私有、会员、登录、年龄限制或验证码保护的视频。
- 不得暗示安装包内置 ASR 权重、云端 LLM key、云端 LLM 模型或用户私有配置。
- 对应平台的安装包经过验证前，不得宣称该平台已支持。
- 公开文案必须区分本地转写和可选云端 AI 整理。
- 如展示 AI 生成结果，必须标注为示例，并避免使用真实用户内容。

## 实施门控

开始实现或发布宣传站前，必须：

- 在 `docs/product-specs/` 下新增或更新对应 product spec。
- 如果工作涉及前端实现、release metadata、部署或 analytics/privacy 决策，创建 active ExecPlan。
- 执行 `python scripts\validate_agents_docs.py --level WARN`。
- 用桌面和移动端截图验证最终页面。
- 验证链接、下载目标、SEO metadata、键盘导航、对比度和 reduced-motion 行为。
