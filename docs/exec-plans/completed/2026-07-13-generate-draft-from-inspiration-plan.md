**Goal:** Let the `智能提炼` workspace turn a single user-selected `Insight` into a ready-to-use written draft (草稿), grounded by optional summary plus anysearch web search, delivered as a third peer target card `生成文字稿` alongside `要点总结` and `启发灵感`.

**Architecture:** Reuse the existing `retry_insights` pipeline as a third target `draft`, seeded by a single `insight_id` plus the task-local `PreferenceSnapshot` read from disk — no new Tauri command, no FrameQ server changes. The existing `draft_agent.py` agent (anysearch MCP web search + skills + planning + `submit_draft`) consumes a new `build_draft_from_inspiration_prompt`; its `(topic, summary, platform)` signature is rewired to `(insight, preference_snapshot, summary)`. The standalone `--generate-draft-json` CLI / `generate_draft_once` path is retired, with its persistence logic absorbed into `retry_insights_once`'s draft branch. The draft is an independent artifact `ai/draft.md`; failure is attributed only to the draft target via `partial_completed`, leaving other targets intact.

**Tech Stack:** Python worker (Pydantic models, `agent_runtime` ToolLoopAgentRunner, anysearch MCP, `SkillManager`), Rust/Tauri command boundary (`video_processing.rs`, `worker_command.rs`), TypeScript/React frontend (workflow state, worker client, target-card UI), pytest + Vitest + cargo test, sanitized GFM rendering.

---

## Purpose / Big Picture

`智能提炼` today exposes two AI target cards — `要点总结` and `启发灵感`. Users want to extend one selected `Insight` from `启发灵感` into a finished, directly-usable written piece, where the writing is grounded by real-world facts found through web search rather than written from the inspiration question alone.

A backend agent experiment already exists (`worker/frameq_worker/draft_agent.py`: anysearch web retrieval + skills + planning + a `submit_draft` delivery contract), but it runs through a standalone CLI (`--generate-draft-json`) seeded by `(topic, summary, target_platform)`, is not wired into Tauri/UI, and conflicts with the product spec's seed, command, and privacy boundaries.

This change ports the agent route onto the spec's product form: a third peer target card `生成文字稿` seeded by a single `Insight`, triggered through `retry_insights` extended with `target="draft"`, reusing the existing confirmation / quota / error-attribution machinery. The user-visible result is select-seed → confirm → generate → view/copy/export, with the card quietly disabled until a seed is chosen and re-disabled when inspiration is regenerated.

## Decisions

Load-bearing decisions carried over from `openspec/changes/generate-draft-from-inspiration/design.md` (D1–D9). The executor should not re-litigate these.

- **D1 — Keep the agent route (no single-prompt downgrade).** anysearch web retrieval is the agent's differentiating value. The spec's "analogous to `build_summary_prompt`" wording is relaxed to "`build_draft_from_inspiration_prompt` builds the seed prompt that the agent consumes."
- **D2 — Seed = single Insight + PreferenceSnapshot; summary optional grounding.** `run_draft` signature changes from `(topic, summary, platform, env)` to `(insight, preference_snapshot, summary, env)`. `build_draft_from_inspiration_prompt(seed, preference_snapshot, summary=None)` maps `topic`→title direction, `followUpQuestions`→section skeleton, `suitableUse`→form/platform, `matchReason`→anchor, `PreferenceSnapshot`→tone/audience/avoid (missing → no-personalization fallback, never fabricated). `sourceChunkId` is source annotation only — its transcript chunk text never enters the prompt. `summary` (要点总结) is read by the worker from `ai/summary.md` (frontend does not send it); missing → degrade to pure web search, no blocking. **Preference snapshot source (A1):** worker reads `ai/preference-snapshot.json` (written at `target="insights"` generation); **the frontend SHALL NOT send `preference_snapshot` for `target="draft"`**, only `insight_id`.
- **D3 — Extend `retry_insights` with `target="draft"`; retire the standalone CLI.** Draft inherits `insights_generating` phase, supervision, cancellation, `partial_completed` degradation, quota precheck, and error attribution. `RetryInsightsRequest` gains optional `insight_id: int` (required non-empty when target=draft). Retire `generate_draft_once` / `GenerateDraftRequest` / `--generate-draft-json`; keep `resolve_draft_credentials` / `checkout_*` for reuse.
- **D4 — `suitableUse` drives form; skill is the only extension point.** The agent system prompt's "target platform" comes from `Insight.suitableUse`. No backend hard-match of suitableUse→skill. v1: `viral-writer` skill covers 抖音 / 小红书 / 公众号 (note 公众号 ↔ 微信公众号 name alignment in the prompt). Other values (视频号 / b站 / 播客 / 课程·社群 / 内部分享) fall back to LLM general capability. Adding a platform = drop a `SKILL.md` under `worker/frameq_worker/skills/<name>/`, auto-discovered, zero code change.
- **D5 — Quota is flat: checkout-time charge, success-independent, no refund; retrieval is free (v1).** Verified against server (`consumeLlmQuota` at `prismaStore.ts:260` / checkout endpoint `server.ts:439` charges immediately, no refund, no success-gating; only idempotency is `(userId, requestId)`) and tauri (`account.rs:314` mints a new `format!("llm-{}", Uuid::new_v4())` per `retry_insights_blocking`). **Model (user choice c): each confirmed generation (including retries) costs exactly 1 quota, regardless of success or failure, non-refundable.** The confirmation page shows a fixed "1". Server is NOT modified. The spec's prior "1 quota = 1 LLM call / by actual call count" wording is withdrawn. anysearch retrieval does not consume user quota.
- **D6 — Privacy boundary relaxed internally; confirmation page unchanged.** The user-visible confirmation-page data notice stays as "send selected inspiration fields + preference snapshot to cloud LLM; do not upload video/audio/full transcript" — no new anysearch disclosure row. Internally, the spec allows the agent to retrieve and feed anysearch results into LLM context; transcript/video/audio/source-URL are still never sent. Boundary A ("no full transcript upload") holds; boundary B ("only") is relaxed.
- **D7 — Draft artifact and state projection.** Write `{task_dir}/ai/draft.md` (`TaskPaths.draft_path` already exists). `ProcessResult` (worker) and `ProcessVideoResult` (Rust) gain `draft: str`. Manifest gains `draft_path` / `has_draft` / `draft_seed_insight_id`; artifact candidate list gains `draft`. During generation, reuse the `insights_generating` phase with `activeAiTarget="draft"`; local transcript stays readable, editing/saving disabled. Failure attributes to draft target only; other targets unaffected on `partial_completed`.
- **D8 — Seed selection and invalidation.** Each Insight in the `启发灵感` list offers a "选为文字稿种子" single-select; selecting writes `draft_seed_insight_id`. Regenerating `启发灵感` successfully clears `draft_seed_insight_id`, returning `生成文字稿` to a quiet disabled state. If `insight_id` is not in `insights.json`, return `DRAFT_SEED_INVALID` and require re-selection — never silently use another entry.
- **D9 — Error stage attribution.** Draft failures use `WorkerError.stage = JobStage.DRAFT_GENERATING` (enum already exists) so attribution lands on the draft target, not insights. UI progress phase still reuses `insights_generating` (`ProcessPhase`); target attribution is projected via `activeAiTarget` / `aiErrorTarget="draft"`, and the UI does not infer target from stage copy.

**Standalone CLI removal:** `generate_draft_once` / `GenerateDraftRequest` / `parse_generate_draft_request` / `--generate-draft-json` are fully retired from production code (grep-verified zero references); credential/checkout lives inside `draft_agent.run_draft._run_async`, so the draft branch resolves it identically with no new checkout path.

**Residual risks / follow-ups:**
- **Cross-session seed persistence (new):** the frontend holds `draftSeedInsightId` in `WorkflowState` only; `HistoryDetailResponse` / `historyItemToWorkerResult` do not surface `draft_seed_insight_id`, so a task reload won't re-show the selected seed until the worker writes it to the manifest on draft generation. Full restore needs a Rust `HistoryDetailResponse` field + history-client threading (flagged in-code at `workflowState.ts:119-124`). In-session selection works fully; worker manifest mirror survives on disk.
- **Manual desktop walk-through (7.4):** executed — seed→confirm→generate→view/copy/export + regenerate→disabled path run end-to-end on a running Tauri desktop app with a completed transcript task; resolved, no longer a release blocker.
- **R2 (web citations):** draft prompt instructs explicit source annotation; structured web-citation model deferred (design R2).
- **R3 (platform form):** `viral-writer` covers 抖音/小红书/公众号 as a soft guarantee (LLM self-loads); other `suitableUse` values fall back to LLM general capability; adding a skill = drop a `SKILL.md`.
- **R1 (server-side cost):** accepted per D5 (user choice c); bounded by `FRAMEQ_DRAFT_MAX_TURNS`.

**Execution-time decisions beyond D1–D9:** (1) `DRAFT_SEED_UNSET` sentinel in `task_store.write_task_manifest` to preserve the frontend-owned seed across manifest rewrites without polluting `ProcessResult`/the worker contract. (2) `draft_generating` added to the `WorkflowStage` union as an error-only stage (documented; excluded from `cancellingFromStage` and all live progress paths) so the worker's error stage type-checks while the UI keeps `insights_generating` as the draft progress phase. (3) The Rust worker command was extended to pass the server source, anysearch checkout URL, and desktop session token alongside LLM checkout, since draft's agent needs the separately-scoped anysearch checkout (recorded in Surprises & Discoveries).
