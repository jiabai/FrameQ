# FrameQ v0.2.16 Open-Source Release Plan

> This ExecPlan is a living document. The sections Progress, Surprises & Discoveries, Decision
> Log, and Outcomes & Retrospective must be kept up to date as work proceeds.

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this
> plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Produce and inspect a Draft `v0.2.16` GitHub Release for Windows x64, macOS Intel, and
macOS Apple Silicon under the approved personal-development/small-user/open-source distribution
policy, then publish it only after a separate final confirmation.

**Architecture:** Release preparation changes metadata, tests, and documentation only; it does not
change FrameQ runtime behavior. One repository-local drift checker makes the desktop version
sources agree. The existing tag-triggered Desktop Release workflow remains the sole artifact
builder and creates a draft before any stable publication.

**Tech Stack:** Node.js test runner, npm, Rust/Cargo, Tauri v2, GitHub Actions, GitHub CLI, Markdown.

---

## Purpose / Big Picture

Users should receive a coherent v0.2.16 release containing the already-validated FrameQ desktop
changes. Windows continues to use the signed Tauri updater path. The two macOS DMGs remain ad-hoc
signed and not notarized, which is acceptable only for the approved personal/open-source audience
with prominent Gatekeeper instructions. No Linux, payment integration, real LLM call, model weight,
credential, or user data enters the release.

## Surprises & Discoveries

- Evidence: `.github/workflows/desktop-release.yml` sets `RELEASE_DRAFT` to true for every tag push,
  so pushing `v0.2.16` creates inspectable artifacts without immediately changing the latest stable
  release.
- Evidence: the macOS jobs already build architecture-specific app bundles, import the bundled
  Python worker/runtime, execute bundled Deno, reject bytecode mutation, and run
  `codesign --verify --deep --strict` before uploading each DMG.
- Evidence: the current version is duplicated in `app/package.json`, two top-level entries in
  `app/package-lock.json`, `app/src-tauri/Cargo.toml`, the `app` package entry in
  `app/src-tauri/Cargo.lock`, and `app/src-tauri/tauri.conf.json`; no current drift test protects
  these sources.
- Evidence: the release workflow requires architecture-specific macOS runtime/media archive secrets
  and the existing Tauri updater signing secrets, but no Apple Developer ID or notarization secret.

## Decision Log

- Decision: Publish v0.2.16 under the personal-development, small-user, and open-source stable
  release policy rather than as a prerelease. Rationale: the user selected option A and accepted the
  one-time Gatekeeper workflow. Date/Author: 2026-07-12 / User + Codex.
- Decision: Keep macOS ad-hoc signing and explicitly disclose the absence of notarization.
  Rationale: this avoids paid Apple program requirements while preserving bundle-integrity checks;
  it must not be represented as Apple identity verification. Date/Author: 2026-07-12 / User + Codex.
- Decision: Create a Draft Release from the tag and stop before publication for artifact review and
  final confirmation. Rationale: publishing changes updater-visible external state, while a draft
  permits safe inspection and correction. Date/Author: 2026-07-12 / User + Codex.
- Decision: Add one narrow version-drift checker rather than introduce a release-management
  dependency. Rationale: five text/JSON metadata sources are simple to validate with Node's standard
  library and existing test runner. Date/Author: 2026-07-12 / Codex.

## Outcomes & Retrospective

The local v0.2.16 release candidate now has consistent version metadata and an automated drift
guard. Local validation passed scripts 13/13, App 35 files / 256 tests, App production build,
Windows Rust 104/104, worker 231/231, Ruff, server 12 files / 57 tests, server build, and governance
validation. The final commit, tag, hosted workflow run, uploaded asset names/sizes/hashes, and
publication state will be added only after those events occur.

Residual risk: macOS users will see a one-time Gatekeeper block because the DMGs are ad-hoc signed
and not notarized. Hosted Apple Silicon validation cannot prove behavior on every clean physical
Apple Silicon Mac. GitHub updater delivery in mainland China remains an accepted v1 test waiver.
