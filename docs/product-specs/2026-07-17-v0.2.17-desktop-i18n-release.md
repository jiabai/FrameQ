# FrameQ v0.2.17 Desktop i18n Release Spec

## Status

Approved for release preparation on 2026-07-17. On 2026-07-22, broad consumer publication was
additionally blocked on the reliability requirements in
`docs/product-specs/2026-07-22-release-reliability-hardening.md` plus the active server
authentication/quota and production-operations ExecPlans. Tag creation and public publication
remain separate approval checkpoints after those blockers close.

## Release Intent

FrameQ v0.2.17 is a stable desktop release for the existing personal-development, small-user, and
open-source distribution model. It packages the completed desktop localization and AI output-language
work as one compatible desktop/worker release while preserving the local-first runtime, lightweight
installer, entitlement, SourceIdentity, and disabled-WeChat-payment boundaries.

## Included Scope

- Bundle Simplified Chinese, Traditional Chinese, and US English UI resources, with a persisted
  `system` language preference and deterministic English fallback for unsupported system locales.
- Localize app-owned settings, account, History, transcript, AI workspace, progress, accessibility,
  known errors, and safe technical-detail surfaces.
- Freeze the resolved UI language when the user confirms summary or Inspiration generation and send
  that closed language enum through the strict desktop-worker contract.
- Ship the structured progress/error event contract together with the matching bundled worker.
- Include the completed `AI整理` to `智能提炼` workspace naming change and Node.js 24-capable GitHub
  Actions maintenance.

## Excluded Scope

- Local video/audio file import remains an unimplemented active plan and is not advertised as a
  v0.2.17 capability.
- Inspiration-to-draft generation and the standalone marketing site remain design/documentation work.
- No new language, automatic AI-output detection, translation, retry, provider call, or AI Credits
  charge is introduced.
- No ASR model weight, LLM credential, payment credential, user configuration, log, task, or user
  artifact is added to an installer.

## Supported Artifacts

- Windows x64 NSIS installer, updater signature, and `latest.json`.
- macOS Intel (`x86_64`) DMG.
- macOS Apple Silicon (`aarch64`) DMG.
- No Linux artifact or Universal macOS binary.

## macOS Trust Boundary

The v0.2.16 ad-hoc-signing policy remains unchanged. Both macOS app bundles must pass bundled-runtime,
Deno, and `codesign --verify --deep --strict` checks on GitHub-hosted macOS runners, but they are not
Developer ID signed or notarized. Release notes must place the Gatekeeper disclosure before download
links and must not imply App Store review, notarization, or silent enterprise deployment.

## Release Flow

1. Synchronize all five desktop version sources at `0.2.17` and keep an automated drift test.
2. Add a real app lint command that performs TypeScript no-emit checking and the production i18n
   literal boundary check without introducing a new lint framework during release preparation.
3. Run the complete local release gates on the release-prep diff and record exact evidence.
4. Retain the accepted atomic authoritative persistence boundary, implement and accept worker
   watchdog plus both server hardening ExecPlans, then rerun the complete release suite on their
   combined reviewed commit.
5. Commit and push the verified release changes only after review, then create annotated tag
   `v0.2.17` as a separate action.
6. Let the tag-triggered Desktop Release workflow create a Draft Release and build all three platform
   artifacts from one tagged commit.
7. Inspect artifact names, updater metadata, bundled runtime imports, Deno smoke, macOS codesign output,
   and release notes before publishing the Draft as a stable non-prerelease release.

## Acceptance Criteria

- `app/package.json`, `app/package-lock.json`, `app/src-tauri/Cargo.toml`,
  `app/src-tauri/Cargo.lock`, and `app/src-tauri/tauri.conf.json` agree on `0.2.17`.
- App tests, lint, production build, Rust tests/rustfmt, worker tests/Ruff, server tests/build, script
  tests, documentation validation, and diff checks pass.
- A Tauri no-bundle release build succeeds with a freshly synchronized packaged worker.
- The Draft Release jobs succeed for Windows x64, macOS Intel, and macOS Apple Silicon, and every
  uploaded artifact reports version `0.2.17`.
- macOS release acceptance records native locale/layout status truthfully; unavailable manual evidence
  remains an explicit residual risk rather than being inferred from shared tests.
- Atomic persistence failure-injection/recovery evidence is recorded in its completed ExecPlan.
  Worker watchdog race/process-tree and available native Windows/macOS evidence must pass and be
  recorded in its active ExecPlan.
- Server OTP/ticket/quota concurrency and production operations evidence is accepted through
  `docs/exec-plans/completed/2026-07-22-server-auth-quota-concurrency-hardening-plan.md` and
  `docs/exec-plans/active/2026-07-22-server-production-operations-hardening-plan.md`; desktop-local
  tests do not substitute for either gate.

## Residual Risk

Real LLM providers may ignore best-effort output-language instructions. FrameQ does not detect or
translate mismatched output and does not spend another AI Credit automatically. macOS Gatekeeper
manual approval remains required until Developer ID signing and notarization are adopted. The existing
main-bundle size warning and external video-platform availability remain non-blocking known limits.
Server authentication/quota and production operations remain release blockers. Atomic persistence
and the worker watchdog are accepted; unavailable macOS/Unix and real-Tauri persistence evidence
plus unavailable macOS watchdog runtime evidence remain recorded residual risks.
